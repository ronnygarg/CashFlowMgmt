Build a polished but highly modular Python analytics dashboard project for electricity prepayment and consumption analysis.

## Core rules
- Do not fabricate or force a merge key between consumption and vend datasets.
- Keep the base directory configurable.
- Respect the existing starter files:
  - requirements.txt
  - config/app_config.yaml
  - README.md
- Keep code modular and easy to refactor.
- Make the Data Quality page prominent.
- Treat vend datetime parsing as provisional if issuedate is incomplete or time-only.

## Implementation style
- Prefer reusable functions over monolithic scripts.
- Include comments, docstrings, and clear TODOs for future combined analysis.
- Do not hide data limitations.

## Objective
Create a Streamlit-based multipage dashboard project that ingests CSV files from a configurable project folder, profiles the data, converts clean standardised versions to Parquet, and presents exploratory analysis pages for:
1. Consumption data
2. Vend / recharge data
3. Data quality assessment
4. Future combined analysis placeholder

This is an early-stage exploratory dashboard for project stakeholders. The design must prioritise:
- flexibility,
- easy restructuring,
- modular code,
- reusable functions,
- clear assumptions,
- future extensibility.

Do NOT over-optimise for the current month only. Build it so future monthly files can be added with minimal code change.

## Important business constraints
1. There is currently NO common key between consumption and vend datasets.
   - consumption appears keyed by `mtrid`
   - vend appears to contain `servicepointno` and `meterno`
2. Therefore, DO NOT attempt a forced merge between the two datasets.
3. Build a placeholder page for future linked analysis, with TODO notes and architecture hooks for when a mapping table or common key becomes available.
4. Treat vend datetime parsing as provisional because the sample `issuedate` may contain time-only values instead of full datetimes.
5. Include data quality and schema diagnostics as a first-class feature.
6. Leave room to add data cleaning rules later, but do not hard-code aggressive cleaning assumptions now.

## Functional requirements

### 1. Data ingestion
- Read all CSV files from `<base_dir>\raw_data\`
- Automatically identify the single consumption file and all vend files
- Combine all vend CSVs into one DataFrame
- Save processed outputs to:
  - `<base_dir>\data\processed\consumption.parquet`
  - `<base_dir>\data\processed\vend.parquet`
- Include helpful logging or user-facing messages when files are missing or malformed

### 2. Schema handling
- Infer schema from current files, but centralise expected field names in config
- Standardise column names to lowercase snake_case if helpful
- Preserve original source columns where useful
- Validate required columns and report missing columns cleanly

### 3. Date handling
For consumption:
- Parse `midnightdate` safely with day-first logic
- Create useful derived fields such as:
  - date
  - month
  - week
  - day_of_week

For vend:
- Attempt to parse `issuedate`
- If `issuedate` appears to be time-only, preserve it and flag this in data quality results
- Derive whatever safe temporal fields are possible
- Add a field indicating the source file name so file-level windowing can be inspected
- Do NOT invent full datetimes if not present
- Use clear TODO comments for future improvement
